## Projeto de Disciplina (PD)

**Formato.** Grupos de 2 ou 3 pessoas.

**Tema.** Cada grupo deve escolher um aplicativo, site ou sistema que use com regularidade (ex.: iFood, Spotify, Letterboxd, Strava, GitHub) e modelar o banco de dados que sustentaria esse sistema. A pergunta-guia: *como ele foi (ou seria) desenvolvido por baixo do capô?* 

**O que estudar.**
- Quais são as **entidades** principais?
- Como elas se **relacionam** (1:1, 1:N, N:N)?
- Quais **restrições** básicas o domínio impõe (unicidade, obrigatoriedade, integridade referencial)?
- E mais importante: Quais foram as decisões técncias que o time de desenvolvimento tomou ao longo da evolução do app? Essas decisões precisam estar registradas nas migrations.

**Entregas por etapa.**
- **Aula 2:** definição do sistema escolhido + schema inicial (mínimo de 5 entidades com relacionamentos).
- **Aulas seguintes:** evolução incremental do schema (novas entidades, normalização, índices, constraints) — uma migração por mudança.
- **Aula final (09/07/2026):** apresentação de 10 min mostrando o sistema, o schema final e as decisões de modelagem.

**Repositório.**
- Criem uma conta no GitHub e um repositório público para o grupo.
- Cada mudança de schema deve ser uma **migração versionada** (`001_init.sql`, `002_add_reviews.sql`, …) com commits descritivos.
- README com: sistema escolhido, integrantes, diagrama ER e como rodar as migrações.

**Sobre uso de ferramentas de LLM para execução dos trabalhos.** Pessoal, tenho visto o uso dessas ferramentas em basicamente todos os trabalhos (elas deixam rastros). Eu não sou contra o uso das ferramentas, mas vc precisam usar com propriedade. Se a ferramenta sugere que vc escreva X, ENTENDA pq ela sugere X e não Y. SEJA CAPAZ de argumentar a favor de X em detrimento de Y para aquela decisão em particular. Se você usar a ferramenta apenas para gerar conteúdo, você deixará de exercitar o raciocínio crítico das tomadas de decisão. Como já falei inúmeras vezes, é muito fácil escrever os DDLs e DMLs. Difícil é saber quando escrever, quando não escrever. Usem as ferramentas de forma que elas ajudem vocês a tomarem melhores decisões (traga exemplos, peça explicações dos design elaborados e tome VOCÊ as decisões). Não use as ferramentas indiscriminadamente. Em última nota: se você vier para a apresentação apenas para ler o que a LLM escreveu, por favor, nem venha.
