Correção — Atividade MER

O que era esperado

✔ Mínimo 5 entidades

Cada entidade com seus atributos e chave primária claramente identificados.

✔ Entidade fraca ou generalização

Ao menos uma das duas. Entidade fraca: retângulo duplo, relacionamento de identificação.

✔ Cardinalidades definidas

1:1, 1:N ou N:M em todos os relacionamentos, com participação (total/parcial).

✔ Atributos nos relacionamentos

Quando o relacionamento tem propriedades próprias (ex: quantidade em "contém").

Correção — Atividade MER

Erros mais comuns

Atributo virou entidade

Ex: "Endereço" como entidade com um único atributo. Se não tem relacionamentos próprios, é atributo.

Cardinalidade invertida

Um cliente faz muitos pedidos (1:N do cliente para pedido) — não N:1 nem N:M.

Chave primária ausente ou ambígua

Todo atributo (ou conjunto) que identifica unicamente uma entidade deve ser sublinhado.

Entidade fraca sem identificação

Entidade fraca precisa do losango duplo (relacionamento identificador) ligando-a à entidade forte.

Correção — Mapeamento MER → Relacional

Correção — Mapeamento

Estratégia de Generalização

Tabela única (USUARIO)

Uma só tabela com todos os atributos. Colunas específicas ficam NULL para quem não se aplica. Simples, mas menos limpo.

✔ Uma tabela por tipo (adotada)

USUARIO com atributos comuns + CLIENTE e ENTREGADOR com atributos próprios e FK para USUARIO. Mais normalizado.

Como a generalização é total e disjunta, toda linha de USUARIO tem exatamente uma linha filha — CLIENTE ou ENTREGADOR, nunca os dois.

Revisão — Normalização

As Três Formas Normais

1FN

Valores atômicos. Sem grupos repetitivos nem listas em uma célula. Toda relação deve ter PK definida.

2FN

1FN + sem dependências parciais. Todo não-chave depende da chave inteira. Só faz sentido com PK composta.

3FN

2FN + sem dependências transitivas. Nenhum atributo não-chave depende de outro atributo não-chave.

"Cada atributo depende da chave, da chave inteira, e de nada além da chave."

Por que normalizar?

Anomalias da Desnormalização

Anomalia de inserção

Para cadastrar um novo item do cardápio é preciso ter um pedido — mas o item pode existir sem pedido.

Anomalia de atualização

Se o Saboroso mudar de bairro, é preciso atualizar todas as linhas do R1. Esquecer uma cria inconsistência.

Anomalia de exclusão

Se o único pedido do X-Burguer for excluído, perde-se o preço do item — informação que deveria persistir.

A normalização elimina essas anomalias garantindo que cada fato seja armazenado uma única vez.

Passo 1 — Identificar Dependências Funcionais

Dependências Funcionais (DFs)

  • num_pedido → data_pedido, cli_cpf, rest_id  (depende só de parte da PK → parcial)
  • cod_item → item_nome, item_preco  (depende só de parte da PK → parcial)
  • cli_cpf → cli_nome, cli_cidade  (não-chave → não-chave, via num_pedido → transitiva)
  • rest_id → rest_nome, rest_bairro  (não-chave → não-chave, via num_pedido → transitiva)
  • (num_pedido, cod_item) → qtd  (depende da chave inteira → ✔ correto)

Pressione → para revelar cada DF

Passo 2 — 1FN

A tabela já está em 1FN?

✔ Valores atômicos

Cada célula tem um único valor — não há listas como "Belém, Ananindeua" nem grupos repetitivos.

✔ PK definida

A chave composta (num_pedido, cod_item) identifica cada linha unicamente.

⚠ Está em 1FN, mas há dependências parciais — precisamos avançar para a 2FN.

Problema: item_nome, item_preco dependem só de cod_item. E data_pedido, cli_cpf, rest_id dependem só de num_pedido. Nenhum dos dois depende da chave inteira.

Passo 3 — 2FN

Eliminar Dependências Parciais

Extraímos os atributos que dependem de parte da PK para tabelas próprias:

-- depende só de num_pedido → vira PEDIDO
PEDIDO(num_pedido, data_pedido, cli_cpf, rest_id)

-- depende só de cod_item → vira ITEM
ITEM(cod_item, item_nome, item_preco)

-- depende da chave inteira → permanece
PEDIDO_ITEM(num_pedido, cod_item, qtd)
  FK: num_pedido → PEDIDO  ·  cod_item → ITEM

⚠ PEDIDO ainda tem problema: cli_cpf → cli_nome, cli_cidade e rest_id → rest_nome, rest_bairro — dependências transitivas.

Passo 4 — 3FN

Eliminar Dependências Transitivas

Em PEDIDO: cli_cpf e rest_id são não-chaves que determinam outros não-chaves. Isso viola a 3FN.

Cadeia transitiva #1

num_pedido → cli_cpf → cli_nome, cli_cidade
Solução: extrair CLIENTE(cli_cpf, cli_nome, cli_cidade)

Cadeia transitiva #2

num_pedido → rest_id → rest_nome, rest_bairro
Solução: extrair RESTAURANTE(rest_id, rest_nome, rest_bairro)

PEDIDO fica apenas com as chaves estrangeiras — sem dados redundantes de cliente ou restaurante.

Resultado Final — 3FN

Esquema Normalizado — iFood

CLIENTE(cli_cpf, cli_nome, cli_cidade)

RESTAURANTE(rest_id, rest_nome, rest_bairro)

PEDIDO(num_pedido, data_pedido, cli_cpf, rest_id)
  FK: cli_cpf → CLIENTE  ·  rest_id → RESTAURANTE

ITEM(cod_item, item_nome, item_preco)

PEDIDO_ITEM(num_pedido, cod_item, qtd)
  FK: num_pedido → PEDIDO  ·  cod_item → ITEM

✔ Cada fato em um único lugar. Sem redundância. Sem anomalias de inserção, atualização ou exclusão.

Prévia — SQL

Do Esquema ao Código: DDL e DML

O esquema normalizado vira código SQL diretamente. Na próxima aula (após a prova) veremos isso em detalhes:

-- DDL: definir a estrutura (CREATE TABLE)
CREATE TABLE cliente (
  cli_cpf    VARCHAR(11) PRIMARY KEY,
  cli_nome   VARCHAR(100) NOT NULL,
  cli_cidade VARCHAR(80)
);

CREATE TABLE pedido (
  num_pedido  SERIAL PRIMARY KEY,
  data_pedido DATE NOT NULL,
  cli_cpf     VARCHAR(11) REFERENCES cliente(cli_cpf)
);

-- DML: inserir dados (INSERT INTO)
INSERT INTO cliente VALUES ('111', 'Ana', 'Belém');
